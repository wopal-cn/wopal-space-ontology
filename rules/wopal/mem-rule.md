# 记忆管理规则

## 核心原则

只沉淀与空间优化、项目建设**高度相关**且具有**长期复用价值**的信息。

## 写入前确认规则

无论通过 `memory_manage` 工具或直接编辑 `MEMORY.md`/`memory/diary/`，任何记忆写入操作：

1. **必须先完整展示准备记录的内容**
2. **必须等待用户明确同意**
3. **严禁未经展示和确认直接写入**

适用于：主动沉淀、用户触发（"记下来"）、蒸馏生成。

## 不记录

- 通用知识（已知常识、可从文档/网络直接获取）
- 临时状态（版本号、当前配置值）
- 会话流水（做了什么、决策过程）
- 单纯的错误描述（无根因、无方法）

## 分类体系

| 分类 | 标识符 | 默认权重 | 判断标准 |
|------|--------|----------|---------|
| 用户要求 | `requirement` | 0.95 | 用户明确要求必须遵守的规则/行为 |
| 用户档案 | `profile` | 0.90 | 用户个人身份和基本信息 |
| 避坑方法 | `gotcha` | 0.85 | 踩过的坑、不要这样做（反面教训） |
| 经验 | `experience` | 0.85 | 验证有效的最佳实践（正面总结） |
| 偏好 | `preference` | 0.80 | 用户倾向但非强制 |
| 事实 | `fact` | 0.75 | 空间/项目特有的客观事实（通用可查的不记录） |
| 知识 | `knowledge` | 0.70 | 空间/项目特有的技术理解（通用可查的不记录） |

区分要点：`preference` 是"倾向"，`requirement` 是"必须"；`gotcha` 是反面教训，`experience` 是正面实践。

## 写入格式（结论前置七原则）

1. **第一句话就是规则本身**：不要用"关键纠正""本次违反""背景："这类元叙述开头。第一行直接说规则，读完即懂，背景放后面
2. **结论前置**：第一行必须是结论/规则/核心内容，背景和细节放后面
3. **直接描述**：不要 `## [类别]:` 前缀（类别由字段或标题标识）
4. **格式简洁**：用普通文本，不要 `**粗体标签**：` 格式
5. **背景必要**：包含"为什么这很重要"或"在什么场景下有用"
6. **内容完整**：保留必要的代码/路径/命令
7. **不截断**：保持完整，去除冗余描述

## 工具参数规范

| 字段 | 参数名 | 说明 |
|------|--------|------|
| 分类 | `category` | 7 个合法分类之一 |
| 正文 | `text` | 遵循结论前置七原则 |
| 重要性 | `importance` | 根据分类默认权重调整 |
| 项目 | `project` | `wopal-space`（空间级）或具体项目名 |
| 标签 | `tags` | 按标签设计规范填写 |

## 标签设计规范

标签在两个环节起作用：用户搜索时拼入 FTS 查询词；自动注入时通过 conceptBoost 提权。每个标签提交前必须答出"它会命中什么场景的搜索"。

### 选择规则

1. 用检索场景词，不用内容碎片——`委派` 是场景，`Done` 是碎片；`验证纪律` 是场景，`checkbox` 是碎片
2. 英文标签优先（conceptBoost 只对英文词生效，每个查询词匹配 tag 子串 +0.05，上限 +0.15）；中文标签兜底（仅参与 FTS 文本搜索，无 boost）
3. 禁止拼接造词（`context-manage-detail` 没人搜），禁止宽泛词（"操作""处理"）
4. 标签数量 ≤5 个——超了说明正文焦点不清晰

### 写入前自检

- 每个标签能说出一条真实检索场景（"搜什么会命中这个标签"）
- 英文标签覆盖所有核心概念（是 boost 主力通道）
- 无内容碎片、无拼接长词、无宽泛词

## 沉淀位置

| 信息性质 | 存储位置 |
|----------|----------|
| 长期知识/经验/避坑 | LanceDB（`memory_manage`），不可用时降级到 `MEMORY.md` |
| 用户偏好 | `USER.md` |
| 工作规则/约定 | `AGENTS.md` |
| 项目特定知识 | 项目 `AGENTS.md` |
| 核心行为特质 | 灵魂（系统提示词） |
| 短期会话记忆 | `memory/diary/` |

## 沉淀决策流程

### 触发时机

- 用户说"记下来/记录知识" → 提取关键事实写入
- 发现重复出现的信息 → 主动提醒沉淀价值，等确认后执行
- 用户偏好含"应该/必须" → 提炼到宪法或 `requirement`

### 逐条筛选

每条候选依次通过以下判断，未通过的在展示时标注原因：

1. **重复检查**：与已有记忆重复 → 跳过或合并
2. **通用性过滤**：通用可查知识 → 跳过
3. **时效性判断**：已修复/已完成问题 → 跳过
4. **会话流水过滤**：调试/决策过程 → 跳过
5. **价值检验**：未来 Agent 读到能避免错误或做出更好决策吗？不能则跳过
6. **格式合规**：符合结论前置七原则，不合规则修正

### 展示与决策

每条候选附筛选建议：

```
[0] <候选内容>
   → 筛选建议：保留/跳过/合并 — <理由>
```

展示完毕后等待用户选择（全写/精选/取消）。严禁跳过展示和确认直接写入。
