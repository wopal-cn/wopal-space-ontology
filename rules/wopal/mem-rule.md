# Memory Management Rules

## Core Principle

Only record information that is **highly relevant** to space optimization and project development, with **long-term reuse value**.

## Pre-Write Confirmation Rules

For any memory write operation, whether via the `memory_manage` tool or direct editing of `.wopal-space/memory/MEMORY.md` / `.wopal-space/memory/diary/`:

1. **Full content must be displayed for review before writing**
2. **Explicit user consent is required**
3. **Writing without display and confirmation is strictly forbidden**

Applies to: proactive recording, user-triggered ("记下来"), and distillation.

## Memory Display Format Specification

Before adding, updating, or deleting a memory, the complete content must be displayed using the following format and confirmed by the user. Display must use **markdown list** format with one field per line. Tables, code blocks, and plain-text summaries are not allowed.

### Required Fields

- **category**: One of the 7 valid categories
- **project**: `wopal-space` or a specific project name
- **importance**: A float between 0 and 1
- **tags**: Comma-separated
- **text**: Full text — no truncation, summarization, or omission

### Supplementary Information

- **dedup result**: State whether the entry conflicts with or complements existing memories, or "no duplicates" if none found
- **tool output**: The complete output returned by `memory_manage add` / `update` / `delete` (including ID, timestamp, etc.) must be displayed verbatim to the user

### Example

```
category: knowledge
project: wopal-space-ontology
importance: 0.95
tags: skill-scripts, ellamaka, restart, hot-reload, verification
text: 技能脚本变更即时生效，不需要重启 ellamaka...

dedup result: no duplicates. Complements [5397c940] (which covers plugin restart requirements)
```

### Distillation Preview

For candidates generated by `memory_manage distill`, each candidate must also be displayed using the format above. After displaying all candidates, wait for the user to decide (write all / select / cancel). Only call `confirm` after explicit user approval.

### Common Mistakes

- ❌ Showing only ID or summary → user cannot evaluate the full content
- ❌ Using table format → long text renders poorly in tables
- ❌ Not displaying the full tool output → user cannot see ID, timestamp, and other metadata

## Do Not Record

- General knowledge (common knowledge, retrievable from docs or web)
- Transient state (version numbers, current config values)
- Session logs (what was done, decision processes)
- Plain error descriptions (no root cause, no methodology)

## Category System

| Category | Identifier | Default Weight | Criteria |
|---|---|---|---|
| User Requirement | `requirement` | 0.95 | Rules/behaviors the user explicitly requires to be followed |
| User Profile | `profile` | 0.90 | User's personal identity and basic information |
| Gotcha | `gotcha` | 0.85 | Pitfalls encountered, things not to do (negative lessons) |
| Experience | `experience` | 0.85 | Verified best practices (positive takeaways) |
| Preference | `preference` | 0.80 | User inclination, not mandatory |
| Fact | `fact` | 0.75 | Objective facts specific to the space/project |
| Knowledge | `knowledge` | 0.70 | Technical understanding specific to the space/project |

Key distinctions: `preference` is inclination, `requirement` is mandate; `gotcha` is a negative lesson, `experience` is a positive practice.

## Write Format (Conclusion-First Six Principles)

1. **Conclusion first**: The first sentence is the rule itself — understood on reading. Do not begin with meta-narrative like "key correction", "this violation", or "background:". Place background and details after the conclusion.
2. **Direct description**: Do not use `## [Category]:` prefixes (category is identified by field or heading).
3. **Plain formatting**: Use plain text. Do not use `**bold label**:` formatting.
4. **Background when necessary**: Include "why this matters" or "in what scenario this is useful".
5. **Content completeness**: Keep necessary code, paths, and commands.
6. **No truncation**: Keep content intact, remove only redundant descriptions.

## Tag Design Specification

Tags serve two purposes: user search terms join the FTS query; auto-injection receives conceptBoost weighting. Before submitting each tag, answer: "What search scenario will hit this tag?"

### Selection Rules

1. Use retrieval scenario words, not content fragments — `委派` is a scenario, `Done` is a fragment; `验证纪律` is a scenario, `checkbox` is a fragment.
2. English tags take priority (conceptBoost only applies to English words; each query word matching a tag substring gets +0.05, cap +0.15). Chinese tags serve as fallback (FTS text search only, no boost).
3. No concatenated coinages (`context-manage-detail` — nobody searches for this). No overly broad words ("操作", "处理").
4. Max 5 tags — more means the focus is unclear.

### Pre-Write Self-Check

- Each tag can articulate a real retrieval scenario ("what query would hit this tag?")
- English tags cover all core concepts (they are the primary boost channel)
- No content fragments, no concatenated long words, no broad words

## Storage Locations

| Information Type | Storage Location |
|---|---|
| Long-term knowledge / experience / gotchas | LanceDB (`memory_manage`); fallback to `.wopal-space/memory/MEMORY.md` |
| User preferences | `.wopal-space/memory/USER.md` |
| Space work rules / conventions | `.wopal-space/REGULATIONS.md` |
| Project or module-specific rules | `AGENTS.md` at project root or in current directory |
| Core behavioral traits | Agent soul (system prompt) |
| Short-term session memory | `.wopal-space/memory/diary/` |

## Recording Decision Flow

### Triggers

- User says "记下来" / "记录知识" → extract key facts and write
- Recurring information detected → proactively suggest recording value, execute after confirmation
- User preferences contain "应该" / "必须" → extract to `requirement`

### Candidate Screening

Each candidate passes through the following checks in order. Those failing a check are annotated with the reason:

1. **Duplicate check**: Already exists → skip or merge
2. **Generality filter**: General retrievable knowledge (see "Do Not Record") → skip
3. **Timeliness check**: Already fixed / completed issue → skip
4. **Session log filter**: Session process log (see "Do Not Record") → skip
5. **Value check**: Will a future agent avoid mistakes or make better decisions by reading this? If not → skip
6. **Format compliance**: Conforms to the Conclusion-First Six Principles; correct if not
