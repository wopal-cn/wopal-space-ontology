# 空间使用指南 — Coding 类型参考

本文档只包含 coding 类型空间相对通用的增量内容。通用内容见通用参考。

---

## 这个空间适合做什么

本空间面向软件开发。主要工作目录是 `projects/`，代码仓库放这里。大部分任务会涉及阅读、编写、审查或管理代码。

---

## 开发工作流程

日常工作走 Issue/Plan 驱动流程：

```
创建 Issue → 出 Plan → 审批 → 实施 → 审查 → 验证 → 归档
```

- **复杂任务**会自动进入 dev-flow 生命周期（Agent 自行判断）
- **简单改动**（改 typo、改配置、单文件修改）可以直接处理
- **所有代码变更**都会经过审查才能完成

你不需要掌握完整的流程。说"处理这个 Issue"或"改这个"就行，我来推进。

---

## 项目级规则

`projects/` 下的每个项目都可以有自己的 `AGENTS.md`，定义项目专属的开发规范：

- 语言和框架约定
- 测试要求
- 提交风格
- Agent 在该项目中的行为

**在某个项目开工前，先看项目的 `AGENTS.md`。** 空间规则是基线，项目规则更具体，冲突时以项目规则为准。

---

## Coding 空间常用命令和技能

| 命令/技能 | 为什么这里更常用 |
|-----------|----------------|
| `dev-flow` | 主要工作流程，自动加载 |
| `agents-collab` | 复杂代码任务拆解为 Wopal→Fae→Rook 流水线 |
| `/review` | 所有非 trivial 代码都要经过审查 |
| `/commit` | 确保提交信息遵循规范 |

**编码任务的协作方式：**

1. Wopal 理解需求、出 Plan
2. Fae 实现代码、跑测试
3. Rook 审查实现质量
4. 发现问题 → Fae 修复 → Rook 复审（最多 3 轮）
5. 全部通过后，Wopal 提交结果并等待你确认

你不需要管理这个流程——告诉我需要做什么就行。
