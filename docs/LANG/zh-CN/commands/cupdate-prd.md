# 创建或更新 PRD

创建或更新产品 PRD 文档。

**用户输入参数**：`$ARGUMENTS`

**参数说明**: 产品名。用户未输入时从 `docs/products/` 查目录匹配推断。

---

## 核心原则

### 输出语言

按用户偏好语言生成或更新文档，除非用户明确要求其他语言。

### 文档路径

**产品 PRD**：使用已有的产品文档命名约定（若已建立）。默认路径：`docs/products/<产品名>/PRD-<产品名>.md`。可接受的已有变体：`PRD.md`、`PRD-*.md`。更新时保留已有路径。

### 上下文收集

读取足够的上下文，避免凭空编造需求。

**必需**：

- 已有目标 PRD（若存在）
- 相关 DESIGN 文档（若存在）
- 当前会话上下文：用户需求、决策、研究结论和未解决的问题

**从实现更新时**：检查代码或项目文档，仅提取产品事实、当前能力和实际边界。不要将实现细节变成 PRD 内容。

**WopalSpace 特定上下文**：在 WopalSpace 内工作时，优先使用规范启动文件和结构文件：

- `.wopal-space/STRUCTURE.md`
- `.wopal-space/REGULATIONS.md`

### 撰写规则

- PRD 回答：构建什么、为谁构建、为何重要、服务什么产品结果。
- PRD 不得解释内部架构、API、存储模式、实现步骤或编码规范。
- 产品 PRD 负责：愿景、用户、产品形态、能力边界、治理和演进。
- 能力范围 / 核心能力边界章节只能描述目标态能力边界：拥有的目标能力、排除的能力和委派边界。
- 能力范围 / 核心能力边界章节不得包含阶段时间、当前/未来分组、实现状态、交付进度、模块状态、复选框或"已完成 / 部分 / 待定"标签。实现状态属于 Phase、Plan、UAT 或 Verification 文档，不进入 PRD 或 DESIGN。
- 不要在 PRD 中包含独立的成功标准或验证信号章节。如需验证信号，放入 Plan、UAT、验证文档或路线图阶段验收说明中。
- PRD 正文必须使用产品语言，而非文档撰写语言。不要说明某个章节的作用、文档如何组织或模板如何使用。
- 避免不能传达产品价值的抽象分类或防御性措辞，如"这不是 X 而是 Y""暴露接口""已成长为""本章节描述"。
- 每个段落和表格行应传达一个产品事实：用户问题、产品角色、用户收益、拥有的能力、排除的边界、产品入口或路线图结果。
- 保留所需的 PRD 结构，但重写章节内薄弱措辞，而非改变结构来回避措辞问题。
- 已有正确内容应保留和精简，而非为新颖而重写。
- 过时内容在有明确证据时应修订或删除。
- 未解决的疑问应标记为待确认，而非默默决定。

## 共享文档头部

每个 PRD 标题后应有简洁元数据块：

```markdown
> **状态**: 活跃  
> **更新日期**: YYYY-MM-DD  
> **关联 DESIGN**: `<路径（若已知）>`
```

若文档语言非英文，使用本地化字段标签。

## 第 0 节：变更日志

每个 PRD 应在文档元数据之后、第 1 节之前，包含简洁的变更日志节。

```markdown
## 0. 变更日志

| 日期 | 类型 | 摘要 |
|------|------|------|
| YYYY-MM-DD | 创建 / 更新 | 一行摘要 |
```

规则：

- 仅记录有意义的文档意图、范围、结构或产品需求变更。
- 不要记录拼写修正、仅格式变更或措辞润色。
- 更新时追加一行摘要本次更新。
- 每行摘要限一行。
- 不要将变更日志放在文档末尾。

## 更新模式

更新已有 PRD 时：

1. 保留已有文档路径和标题，除非明显错误。
2. 更新 `更新日期` 日期。
3. 对照以下内容核对文档：
   - 用户确认的需求和会话决策
   - 已实现的代码事实
   - 相关 PRD/DESIGN 文档
4. 当已有结构不完整时，补充缺失的必要章节。
5. 当证据明确时，删除或修订过时声明。
6. 在第 0 节追加一行变更日志。
7. 未解决项明确标记为"待确认"或文档语言等价表述。
8. 从 PRD 中删除独立的成功标准或验证信号章节；不要用更名后的验证章节替代。

不要仅仅因为代码中存在实现内部细节就加进去。将实现事实转化为产品能力和边界。

## 撰写质量标准

PRD 文字必须读起来像产品文档，而非模板说明、架构笔记或实现注释。

推荐模式：

- "用户可以 ..." / "Wopal 可以 ..." / "CLI 提供 ..."
- "此能力降低 ..." / "此保护 ..." / "此保持 ..."
- "CLI 拥有 ..." / "CLI 不拥有 ..."
- "阶段 N 交付 ..."

拒绝并重写的模式：

- 章节注释："本章节描述 ...""根据模板 ...""文档应 ..."
- 模板注释："根据模板 ...""文档应 ..."
- 模糊演进语言："已成长为 ...""成为可扩展入口"
- 无产品价值的抽象对比："不是 API 平台""不是 X 而是 Y"
- 仅架构标签不说明用户收益："暴露产品接口""作为抽象层"

当句子不符合标准时，在定稿 PRD 前重写为具体的产品陈述。

- [ ] 选择了正确的模板：产品
- [ ] 文档语言符合用户偏好
- [ ] 头部包含当前更新日期
- [ ] PRD 保持产品层面，避免架构/实现细节
- [ ] 能力范围 / 核心能力边界仅包含目标态边界，不含阶段时间或实现状态
- [ ] 实现状态不属于 PRD 或 DESIGN，已留给 Phase、Plan、UAT 或 Verification 文档
- [ ] PRD 中无独立的成功标准或验证信号章节
- [ ] 保留所需 PRD 结构同时改进章节措辞
- [ ] PRD 正文不含模板注释、章节说明或文档撰写语言
- [ ] PRD 正文避免模糊演进语言和不能传达产品价值的抽象对比
- [ ] 每个段落和表格行传达具体的产品事实或边界
- [ ] 已有正确内容已保留
- [ ] 过时内容已修订或删除
- [ ] 变更日志已为有意义的创建/更新变更而更新
- [ ] 相关文档已链接

## 完成后的回复

创建或更新 PRD 后，以用户语言回复：

1. 文件路径
2. 创建/更新摘要
3. 有意义的添加、修订、删除/废弃和待确认项
4. 建议下一步：PRD 写完后，通常接着写总体设计（`/cupdate-design`），把产品愿景落地为系统架构
